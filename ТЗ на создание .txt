ТЗ
Сделай проект интеллектуальной веб‑админки для RAG‑системы на Flask. Проект должен быть production‑like по архитектуре, но без лишнего усложнения: простой, модульный, стабильный, расширяемый и пригодный для работы в клиентской среде. Основной интерфейс системы — веб‑админка, через которую можно управлять документами, индексацией, retrieval, качеством ответов, наблюдаемостью и настройками базы знаний.

Проект должен развивать существующий rag_service, а не переписывать его с нуля. Нужно сохранить и улучшить текущую архитектуру: config.py, app.py, ingestion, retrieval, vector_store, шаблоны и статику. Для разработки векторное хранилище должно работать через Chroma, а архитектура должна сразу предусматривать возможность подключения Supabase pgvector как продакшн‑backend без переписывания основной логики.

Цель проекта
Сделай не просто страницу загрузки файлов, а полноценный интеллектуальный центр управления RAG‑системой. Админка должна помогать:

загружать и индексировать документы;

заменять старые версии документов новыми;

удалять документы и очищать коллекции безопасно;

настраивать retrieval и generation;

тестировать качество ответов;

видеть метрики эффективности поиска и ответа;

диагностировать причины плохой работы RAG;

удобно эксплуатировать систему у клиента без ручной работы с папками и файлами.

Основные требования
Используй Flask как основной backend и web UI. Интерфейс должен быть понятным для человека, который не хочет работать напрямую с файловой системой, Chroma, SQL и внутренними каталогами проекта. В админке должны быть безопасные действия, понятные статусы, лог операций и хорошие подсказки.
Архитектура должна быть модульной: backend‑логика отдельно, UI отдельно, конфигурация отдельно, сервисы ingestion/retrieval/evaluation отдельно. Приоритеты проекта — простота, скорость, стабильность и возможность постепенно наращивать функциональность.

Поддерживаемые коллекции
Система должна работать минимум с тремя логическими коллекциями базы знаний:

faq

instructions

contracts

В интерфейсе должно быть удобно выбирать коллекцию при загрузке, поиске, оценке качества и администрировании документов.

Документы и ingestion
Система должна поддерживать загрузку документов форматов:

pdf

txt

docx

md

Нужно реализовать ingestion‑пайплайн, который:

принимает файл через веб‑интерфейс;

сохраняет его в правильную коллекцию;

парсит текст;

режет на чанки;

сохраняет metadata;

отправляет в vector store;

перемещает файл в processed или failed;

логирует результат операции.

Нужно убрать необходимость ручного копирования, удаления и замены файлов в папках. Все основные действия над документами должны выполняться из админки.

Управление документами
Сделай в админке отдельный полноценный раздел управления документами. Он должен включать таблицу документов и операции над каждым документом.

Для каждого документа отображай:

имя файла;

коллекцию;

статус;

дату загрузки;

размер файла;

количество чанков;

document_id;

hash или версию файла;

дату последней индексации.

Для каждого документа реализуй действия:

просмотреть metadata;

просмотреть чанки;

переиндексировать;

заменить новой версией файла;

удалить только из индекса;

удалить файл и запись из индекса;

переместить в другую коллекцию.

Замена документа должна работать как безопасная операция обновления: старые чанки документа удаляются из векторной базы, новый файл загружается, заново индексируется и получает обновлённую metadata. Это должно происходить через UI, без ручной замены файла в каталоге проекта.

Очистка и destructive actions
Сделай безопасные функции очистки, но не ограничивайся одной опасной кнопкой “очистить всё”. Нужны отдельные операции:

очистить только очередь input;

очистить только failed;

очистить конкретную коллекцию;

полностью сбросить индекс;

удалить конкретный документ.

Все destructive actions должны иметь:

явное предупреждение;

текст подтверждения;

отображение количества затрагиваемых документов и чанков;

логирование операции.

Экран дашборда
Сделай главный экран админки как операционный дашборд состояния системы. Он должен показывать:

общее количество документов;

общее количество чанков;

количество документов по коллекциям;

количество ошибок ingestion;

количество документов в input, processed, failed;

последние операции;

статус vector backend;

статус embedding model;

статус LLM;

среднее время retrieval;

среднее время генерации ответа;

end‑to‑end latency;

error rate.

Дашборд должен помогать быстро понять: система работает нормально или нет, где проблема — в ingestion, retrieval, генерации или конфигурации.

Экран индексации
Сделай отдельную страницу индексации и загрузки документов. Она должна включать:

drag & drop upload;

выбор коллекции;

отображение списка файлов в очереди;

preview chunking до индексации;

размер chunk;

overlap;

dry‑run режим;

кнопки индексировать выбранное / индексировать всё;

прогресс обработки;

лог ошибок и результатов.

Пользователь должен видеть, сколько чанков получится, какие metadata будут сохранены и что произошло после индексации.

Экран поиска и тестирования RAG
Сделай отдельный RAG Playground для отладки и демонстрации качества поиска и ответа. Этот экран должен быть одним из самых важных.

На этом экране должны быть:

поле вопроса;

выбор коллекции;

top_k;

temperature;

score_threshold;

режим “только retrieval без LLM”;

режим “показать retrieved chunks”;

режим “показать similarity score”;

режим “показать prompt / system settings”;

итоговый ответ;

список источников;

retrieved chunks;

latency по этапам.

Нужно показывать, как именно сработал поиск: какие chunk’и были найдены, из каких документов они пришли, какие были score, почему был сгенерирован именно такой ответ. Это критично для диагностики качества RAG.

Настройки retrieval и generation
Сделай удобный экран настроек, разделив retrieval и generation на разные смысловые блоки.

Retrieval settings
Нужно поддержать:

top_k;

score_threshold;

max_context_chunks;

max_chunks_per_document;

deduplication on/off;

collection filtering;

search mode: semantic / hybrid / fallback;

rerank on/off;

rerank limit.

Generation settings
Нужно поддержать:

temperature;

max tokens;

модель генерации;

системный prompt;

стиль ответа;

режим “отвечать только по базе знаний”;

fallback “не найдено в базе”;

guardrails против домысливания.

Экран качества и evaluation
Сделай отдельный раздел качества RAG, похожий по смыслу на lightweight‑версию RAGAS dashboard. Важно показывать не только оперативные метрики, но и офлайн‑оценку качества на тестовых вопросах.

Раздел должен позволять:

загружать тестовый набор вопросов;

хранить reference answer;

хранить expected document или expected collection;

запускать batch evaluation;

видеть результаты по каждому кейсу;

видеть агрегированные метрики;

видеть провальные примеры;

экспортировать отчёт.

Минимально включи следующие метрики:

Answer Relevance — насколько ответ соответствует вопросу.

Faithfulness — насколько ответ подтверждается retrieved context.

Context Precision / Context Relevance — насколько retrieval вернул полезный контекст.

Context Recall — хватает ли retrieved context для построения хорошего ответа.

Precision@K и Recall@K для retrieval.

Hit@K.

Latency retrieval / generation / end‑to‑end.

Если полный RAGAS сразу слишком тяжёлый для MVP, реализуй свою внутреннюю evaluation‑прослойку с сохранением структуры данных так, чтобы позже можно было легко подключить RAGAS.

Интеллектуальность интерфейса
Сделай интерфейс не просто административным, а интеллектуальным. Он должен не только показывать настройки, но и объяснять, что происходит, и подсказывать, что можно улучшить.

Добавь:

рекомендации по top_k на основе истории запросов;

предупреждения о слишком длинных chunk’ах;

сигналы о дубликатах в документах;

сигналы о низком retrieval score;

предупреждение, если ответы часто формируются при слабом контексте;

подсказки по quality degradation после изменения настроек;

сравнение конфигов “до / после”.

Наблюдаемость и логи
Сделай полноценную наблюдаемость по этапам RAG. Нужно логировать и отображать:

ingestion events;

parsing errors;

chunking stats;

embedding errors;

retrieval latency;

generation latency;

total latency;

token usage;

error rate;

system health события.

Должен быть экран или блок с последними событиями и фильтрами по типу события.

UX‑модель
Сделай двухуровневую модель интерфейса:

базовый режим — для клиента или контент‑менеджера;

экспертный режим — для разработчика или технического администратора.

В базовом режиме показывай только то, что нужно для ежедневной работы:

документы;

загрузка;

статус;

тест вопроса;

основные KPI.

В экспертном режиме открывай:

chunking settings;

retrieval thresholds;

reranking;

evaluation;

advanced diagnostics;

backend settings;

raw logs.

Архитектурные требования
Разработай проект как расширение текущего rag_service. Не ломай существующую логику, а улучшай её через слои и сервисы.

Структура должна включать минимум:

config.py

app.py

ingestion/

retrieval/

vector_store/

evaluation/

admin/ или services/

templates/

static/

Нужно предусмотреть:

единый config;

автоматическое создание нужных директорий;

безопасную работу с путями от корня проекта;

Chroma backend для dev;

интерфейс адаптера под Supabase/pgvector для prod.

Технические ограничения
Не делай из проекта чрезмерно сложный enterprise‑монстр. Сначала нужен сильный MVP, который уже реально можно использовать. Это соответствует приоритету на простоту, скорость и стабильность.
Пиши production‑like код, но без ненужной абстракции. Если какая‑то часть пока реализуется stub‑ом, оставь для неё понятный интерфейс и комментарий, чтобы позже заменить реализацию без переписывания системы.

Что должно получиться в MVP
На первом рабочем этапе проект должен уже уметь:

загружать документы через UI;

индексировать их по коллекциям;

показывать список документов;

заменять документ новой версией;

удалять документ из индекса;

очищать коллекцию;

тестировать retrieval и answer generation;

показывать retrieved chunks и scores;

настраивать top_k, temperature и базовые thresholds;

показывать основные KPI и логи;

сохранять структуру для дальнейшего подключения расширенной evaluation‑системы.

Что нужно от реализации
Сделай проект поэтапно. Сначала предложи архитектуру и структуру файлов, потом реализуй backend, потом admin UI, потом evaluation и observability. На каждом этапе давай код, который можно сразу вставлять в проект.

Пиши так, чтобы проект можно было развивать без переезда на другую архитектуру. Все решения должны быть совместимы с дальнейшим переходом от Chroma к Supabase/pgvector.

Формат работы
Когда будешь реализовывать проект по этому ТЗ:

сначала предложи структуру проекта;

затем список файлов первого этапа;

потом генерируй код файлами, а не разрозненными кусками;

для каждого этапа объясняй, куда вставлять код;

не переписывай весь проект без необходимости;

сохраняй совместимость с уже сделанными частями
